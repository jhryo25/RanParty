# Project Brief

1. Inspect the workspace structure before opening files. Prioritize manifests, entry points, current documentation, recent outputs, and configuration files.
2. Distinguish verified facts from inference. Cite concrete relative file paths for important claims.
3. Summarize the project using: goal, current architecture, working capabilities, current risks, and recommended next actions.
4. Keep the brief concise. Do not copy secrets, API keys, large source blocks, or unrelated file contents.
5. If the workspace is incomplete or contradictory, state what is missing instead of guessing.
6. Do not modify files unless the user explicitly asks for changes.
