## 四阶六类层级架构总览

```
L0 — 基本指令集（固定前缀，每次会话自动嵌入）
├── SOUL.md      ← 身份锚点 + 输出模式系统 + 决策规则
├── AGENTS.md    ← 运行红线
└── TOOL.md      ← 工具操作指南

L1 — HUB（本文件，初次会话自动加载）
└── HUB.md       ← 中枢：目录树 + 版本管理 + 全生命周期操作

L2 — 技能层（按需加载）
├── L2/Skill/    ← 持久性逻辑闭环（解决问题）
└── L2/Exp/      ← 经验主义使用手册（外部工具）

L3 — 知识层（定点检索）
├── L3/index/    ← 正在积累沉淀的百科级静态知识（增量覆写）
├── L3/project/  ← 当前项目追踪（动态更新，全 md 格式）
└── L3/history/  ← 项目闭包归档（只写不改）

CatTemp/         ← 缓存/中介文件（独立于知识层级）
```

## 📖 L2-Skill — 持久技能

**定位：** 解决某类问题的系统性知识，逻辑闭环，跨项目可复用。生命周期长（6个月以上）。

**严格格式要求：**
```markdown
# <文件名>.md — 简短标题

> 版本：x.y | 最后更新：YYYY-MM-DD
> 类别：L2-Skill
---
## 参考来源
| 时间 | 位置 | 说明 |
|------|------|------|
```

**创建条件：**
- 该知识至少跨越 2 个以上项目仍保持形态稳定
- 有可验证的逻辑规则或规范体系
- 大概率在 6 个月以上仍保持有效

**更新规则：**
- 版本号 +1（小改 x+1，大改 y+1）
- 同步更新 HUB.md 的版本号
- 涉及底层原则的改动需确认

**禁止：**
- 不允许把项目特有的配置写进 Skill（归属 project）
- 不允许把工具踩坑写进 Skill（归属 Exp）
- 不允许把项目叙事/动态状态写进 Skill（归属 L3）

---

## 📖 L2-Exp — 经验文档

**定位：** 对外部工具和服务的使用方法、部署经验、踩坑记录。依赖具体版本和运行环境，生命周期短于 Skill。

**严格格式要求：**
```markdown
# <文件名>.md — 标题

> 版本：x.y | 最后更新：YYYY-MM-DD
> 类别：L2-Exp
---
## 参考来源
```

**创建条件：**
- 引入了新的外部工具/平台/服务
- 有可复现的部署步骤或踩坑记录

**更新规则：**
- 版本号 +1
- 环境变化（软件版本/API 变更）时及时更新
- 过时内容移入 TRASH/ 而非删除

**禁止：**
- 不允许把跨项目持久原则写进 Exp（归属 Skill）
- 不允许把项目叙事写进 Exp

## 📖 L3-Index — 百科级静态知识

**定位：** 积累沉淀的百科级静态知识。增量覆写。

**覆写规则：**
- 对已有文件保持原有信息进行增量覆写——在保留已有内容的基础上追加新信息
- 不追加补充文件——要补充的内容直接写在原文件末尾
- 如需修改文件的层级结构（如重组章节），需经确认
- 错误更正：直接在原文中修正，并在文件头标注 `(YYYY-MM-DD 修正)`

**新建规则：**
- 新增定稿知识时，先在 CatTemp 写草稿
- 确认后再写入 L3/index/
- 文件名格式：`YYYY-MM-DD_描述.md`

**删除规则：**
- 删除前先确认该文件是否被其他位置引用

> **案例 1：** 你刚完成 C# 状态机模式的深入分析 → 在 `CatTemp/` 写草稿 → 确认后写入 `L3/index/CSharp/2026-07-02_状态机模式总结.md`
>
> **案例 2：** 你研究了三个游戏的经济系统设计 → 沉淀到 `L3/index/GameDesign/经济系统模式.md`
>
> **案例 3：** 之前写的 C# 架构分析有新发现 → 直接在原文件末尾追加新段落，标注日期

## 📖 L3-Project — 当前项目追踪

**定位：** 所有活跃项目的动态追踪。统一入口文件为 `project.md`。

**project.md 模板：**
```markdown
# 项目名 — 项目档案

> 类型：<游戏/工具/写作/...>
> 版本：x.y | 最后更新：YYYY-MM-DD
> L2依赖：<引用的 L2-Skill/Exp 文件路径>
> 路径：L3/project/<项目名>/

---

## 项目目的
<一句话说明>

## 项目周期
- **开始**：YYYY-MM-DD
- **当前阶段**：<阶段描述>

## 进度状态
| 模块 | 状态 | 说明 |
|------|------|------|

## 素材清单
<项目相关的原稿/设计/资产文件列表>

## 下一步
<待办事项列表>
```

**更新规则：**
- 每次收工时更新 `project.md` 中的进度状态和下一步
- 项目素材可按需增删
- 项目完结时，将 `project.md` 归档到 L3/history/，并删除 L3/project/ 中对应目录

> **案例 1：** 你在做一款 AVG 游戏 → `L3/project/我的AVG/project.md`，记录剧本进度、美术资产、引擎开发状态
>
> **案例 2：** 你在开发一个 Unity 独立手游 → `L3/project/动作手游/project.md`，追踪 Core 层/外观层/关卡设计
>
> **案例 3：** 项目做到一半搁置了 → 在 project.md 中标注状态为「暂停」，保留所有素材不删，等以后继续

## 📖 L3-History — 项目闭包归档

**定位：** 已完结项目的叙事记录、实验总结、迭代过程。只写不改，入库即定稿。

**写入命名规则：**
- `YYYY-MM-DD_章节名或事件名_类型.md`
- 同一项目的文件按日期自然排序
- 类型标签：`全文留档`、`实验记录`、`迭代总结`、`完结归档`

**总表索引：**
- 每个 L3/history/ 子目录入口处维护一个 `_index.md` 文件
- 格式：`| 日期 | 文件 | 类型 | 摘要 |`

**铁律：**
- 写入后不允许修改——时间是单向的
- 如需更正，追加新的更正记录，不修改旧文件
- 文件名中的日期使用实际发生日期，非写入日期

> **案例 1：** AVG 项目开发完成 → 写 `2026-12-01_AVG项目开发全记录_完结归档.md`，更新 `_index.md`
>
> **案例 2：** 一个实验性功能失败但教训宝贵 → 写 `2026-09-15_物理引擎自实现实验_实验记录.md`，记录为什么失败
>
> **案例 3：** 一次重要的架构升级 → 写 `2026-10-20_从单体到模块化_迭代总结.md`，记录升级前后的对比和决策链

## 📋 收工流程（工作结束规范）

当你说出「收工」或「沉淀」时，执行以下检查表：

### Step 1 — 归档定稿知识到 L3
- **L3/index/**：新增的定稿知识写入（需确认后才可覆写或删除已有文件；覆写时保持原有信息进行增量覆写，不追加补充文件）
- **L3/project/**：更新对应 `project.md` 的进度状态和下一步
- **L3/history/**：叙事记录/实验总结按命名规则写入

### Step 2 — 更新 L2 文件 + HUB
- 更新涉及的 L2 Skill/Exp 文件（版本号 +1）
- 更新 HUB：目录树 + 版本号
- 如有 TRASH 变更，同步更新垃圾桶记录

### Step 3 — CatTemp 清理
- 删除中介/草稿文件

## 📋 新增与更新规则

### 新增 L2-Skill
1. 新建 `.md` 文件到 `L2/Skill/` 下
2. 写入严格格式要求的文件头（版本号、最后更新、类别）
3. **更新 HUB.md：** 在目录树中追加一行
4. 在 `L3/index/` 下创建同名子目录（作为 L3 索引储备）

### 新增 L2-Exp
1. 新建 `.md` 文件到 `L2/Exp/` 下
2. 写入严格格式要求的文件头
3. **更新 HUB.md：** 在目录树中追加
4. 如有定稿知识产生 → 归档到 `L3/index/` 对应子目录

### 更新 L2-Skill
- 批量收集所有改动点 → CatTemp 写完整草稿 → 定稿后再一次覆盖
- 版本号 +1，同步更新 HUB 的版本号
- 涉及底层原则的改动需确认

### 更新 L2-Exp
- 环境变化时及时更新
- 版本号 +1
- 过时内容移入 TRASH/ 而非删除

### 删除/合并 L2
1. 从 `L2/Skill/` 或 `L2/Exp/` 中移除文件（或合入其他文件）
2. **必须同步更新 HUB.md** 的目录树
3. 清理对应的 `L3/index/` 子目录

## 📋 版本号公约

所有 RanParty 知识文件的版本号遵循三级分类：

| 级别 | 触发条件 | 操作 | 示例 |
|:-----|:---------|:-----|:-----|
| **P0** | 内容/逻辑/架构发生实质变化 | 版本号 `x.y` 中的 `y` +1 | 新增章节、修改规则、重写内容 |
| **P1** | 错字修正、格式调整、索引同步 | 版本号不动，日期更新 | Typo 修复 |
| **P2** | 纯外观/装饰性改动 | 什么都不动 | 加空行、调 emoji |

**例外：** HUB 自身的版本号——任何对 HUB 内容的写入都算 P0，因为 HUB 是路由枢纽。
**L3/history/ 不设版本号**，入库即定稿，时间是单向的。
