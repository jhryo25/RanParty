> 版本：2.0-public | 2026-07-02

## 一、使用流程

→ 接到 C# 编码需求（新功能 / 修复 / 扩展）

→ **前置确认（三项齐备才写码）**
  1. 业务背景 — 这段代码解决什么问题、属于哪个模块/项目
  2. 数据处理 — 输入是什么、输出是什么、核心逻辑是什么
  3. 输出范围 — 所有要输出为一个 .cs 文件的内容

→ **遵循核心规则编码**
  - 按 Static C 风格写码
  - 定位归属的静态类
  - 在现有框架上叠加扩展，不重构架构

→ **逻辑自检**
  - 函数签名是否匹配
  - 参数类型和引用是否存在
  - 信号 serial 去重是否需要
  - 序列化字段是否包含分隔符

→ **按渠道交付**
  - 直接贴代码块或写入指定路径

→ **问题已解决，产出已验证**
  → 发出沉淀请求，对接收工流程

## 二、核心规则

### 2.1 Static C 风格
- **逻辑层全部使用静态类**，不使用继承和多态
- 外观层仅做呈现——Core 逻辑不依赖任何 UI 框架组件系统
- 性能优先，信任自己，不信托管抽象

### 2.2 字符串处理
- 手工管字符串格式，不用 JSON/反射/序列化框架
- 能 Split 就 Split，能直接赋值就不调库
- StringBuilder 用于大量拼接，不用 `+` 反复 concatenate

### 2.3 数据访问
- 避免反射：Type.GetProperty/SetValue 有性能开销且失去编译期检查
- 避免 LINQ 滥用：`Where().Select().ToList()` 在热点路径上慢
- 简单循环 > 抽象查询

### 2.4 序列化
- 尽量不用 Newtonsoft.Json / System.Text.Json
- 手动构建字符串格式更可控（自定义分隔符如 `ⓐ`）
- 序列化前验证字段不包含分隔符

### 2.5 内存管理
- 关注 GC 压力：避免在更新循环中 new 对象
- struct 而非 class 适合小块数据传递
- 缓存 array/List 复用而不是反复分配

### 2.6 物理自实现
- 2D 碰撞/物理不使用引擎内置物理，由 Core 层自行计算
- 碰撞检测在逻辑帧中执行，与渲染帧解耦

### 2.7 布尔/枚举位掩码
- 布尔状态和枚举状态优先使用 `int` 位掩码
- 示例：`stateFlags |= 1 << 3`

### 2.8 信号去重（serial 模式）
```csharp
public static int CommandSerial = 0;
public static int PrevSerial = 0;

// 消费方每帧
if (PrevSerial != C_Command.CommandSerial) {
    PrevSerial = C_Command.CommandSerial;
    // 处理指令…
}
```

### 2.9 状态机规范
- 基态 + 模态栈（基态互斥，模态阻塞基态，可堆叠弹出）
- 函数引用寻址：`stateUpdate = UpdateMenu` 直接赋值切换，禁止 `switch(string)`
- 子步骤用 `Action _subStep` 推进

### 2.10 静态类拆分
| 推荐拆分 | 职责 |
|---------|------|
| C_Player | 玩家数据、属性、背包 |
| C_StoryFlag | 剧情开关、进度标记 |
| C_Inventory | 物品系统 |
| C_Map | 地图/坐标逻辑 |
| C_Combat | 战斗计算 |

- 跨类通信通过 Command 缓冲或直接字段访问
- 不引入事件系统或依赖注入

### 2.11 注释与命名
- 所有公开函数写 XML 注释（描述功能，不描述实现细节）
- 动词开头简短命名：`Get`/`Set`/`Load`/`Save`/`Parse`/`Build`

### 2.12 UI 边界协议
- Core 层不引用 UI 框架命名空间下的任何类型
- 外观层通过 Command 缓冲写入 + Core public 字段轮询通信

## 三、行为准则

在该 Skill 下处理问题的逻辑方式——职业的"软技能"

- **不主动推荐 MonoBehaviour 方案** — 架构约束：外观层与核心层分离
- **不引入第三方物理引擎** — 物理系统自实现
- **不推荐事件系统/依赖注入** — 与 Static C 风格冲突（命令缓存 + 直接字段访问替代）
- **代码注释用中文** — 所有注释、回复统一中文
- **扩展在现有框架上叠加，不重构架构** — 除非明确提出
- **前置信息不足时不写码** — 确认业务背景 + 数据处理 + 输出范围后再生成
- **单文件聚合** — 一个新功能输出为一个 .cs 文件，不把无关片段塞进一个文件
